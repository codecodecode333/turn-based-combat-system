# CODE_STRUCTURE.txt v0.2

## 목적

본 문서는 프로젝트의 런타임 코드 구조를 설명한다.
목표는 다음과 같다.

* 현재 전투 코드의 책임 분리를 명확히 한다
* 새 기능 추가 시 어디를 수정해야 하는지 빠르게 판단할 수 있게 한다
* 전투 UX / 실행 로직 / AI / 타일 / 상태이상의 경계를 유지한다
* 향후 리팩토링 시 기준점으로 사용한다

본 문서는 **현재 구현 구조**를 기반으로 하되,
향후 확장 가능한 형태로 레이어를 재정리한다.

---

# 1. 전체 구조 개요

현재 프로젝트는 크게 다음 레이어로 나뉜다.

```plaintext id="fzzkm3"
Presentation Layer
→ Battle Flow Layer
→ Rules / Resolution Layer
→ World / Grid Layer
→ Unit / State Layer
→ Data Layer
→ AI Layer
```

---

# 2. 권장 레이어 구조

## 2.1 Presentation Layer

플레이어가 직접 보는 요소를 담당한다.

포함 대상:

* BattleInput
* TileHighlighter
* UnitHud
* 각종 월드 FX / 상태이상 FX
* 버튼 / 텍스트 / 턴 UI

책임:

* 입력 받기
* 하이라이트 / 프리뷰 표시
* HP/AP/HUD 표시
* 실패 피드백 표시
* 타겟 표시
* 월드 이펙트 표시

이 레이어는 **규칙을 계산하지 않고**,
이미 계산된 결과를 보기 좋게 보여주는 역할에 집중한다.

---

## 2.2 Battle Flow Layer

전투의 진행 순서와 턴 흐름을 담당한다.

핵심 클래스:

* BattleController

책임:

* 턴 순서 관리
* 플레이어 턴 / 적 턴 전환
* 행동 큐 관리
* Confirm / Cancel 흐름
* Move → Skill 실행 순서 제어
* 애니메이션 이벤트와 전투 결과 연결
* Hazard / Status / EndTurn 타이밍 연결

BattleController는 **전투의 오케스트레이터**다.
직접 모든 규칙을 계산하기보다,
각 하위 시스템을 호출하고 연결하는 역할이 중심이다.

---

## 2.3 Rules / Resolution Layer

전투 규칙과 실제 판정을 담당한다.

핵심 클래스:

* CombatResolver
* CombatTargetResolver
* HazardSystem
* SkillMetaUtility (존재 시)
* 기타 전투 규칙 유틸리티

책임:

* 타겟 판정
* 스킬 실행 가능 여부 검증
* AP 소모
* SkillEffect 적용
* Counter 처리
* Hazard 처리
* Friendly Fire / LOS / Range / Target Rule 공유

이 레이어는 **Preview / Player / AI가 공통으로 사용해야 하는 규칙**의 중심이다.

핵심 원칙:

```plaintext id="15wueg"
같은 상황이면
Preview / Execution / AI가
반드시 같은 결과를 반환해야 한다
```

---

## 2.4 World / Grid Layer

전투 맵, 위치, 이동, 점유, LOS를 담당한다.

핵심 클래스:

* GridManager
* TileView
* TileData

책임:

* Grid 좌표 ↔ 월드 좌표 변환
* 점유 상태 관리
* 이동 가능 타일 계산
* 경로 복원
* 이동 루틴
* 타일 조회
* LOS 판정
* Height 이동 제약

이 레이어는 **맵 자체의 법칙**을 담당하며,
전투 규칙 레이어와 연결되지만 분리되어야 한다.

예:

* GridManager는 “갈 수 있는가”를 판단
* CombatTargetResolver는 “맞출 수 있는가”를 판단

---

## 2.5 Unit / State Layer

유닛 개별 상태와 상태이상 라이프사이클을 담당한다.

핵심 클래스:

* Unit
* StatusEffect 계열
* 각 Status 클래스
* ShieldStatus 등 특수 상태

책임:

* HP / AP / 이동력 / 위치 상태
* 사망 여부
* 상태이상 보유
* 상태이상 Apply / Tick / Remove
* 피격 / 힐 / 실드 / 무적 처리
* 애니메이션 이벤트 발생

Unit은 “전투 오브젝트”이며,
상태이상은 Unit 내부 상태로 붙는다.

---

## 2.6 Data Layer

에디터에서 정의되는 설계 데이터를 담당한다.

핵심 클래스:

* SkillData
* SkillEffect
* TileData
* AIProfile

책임:

* 스킬 정의
* 타겟 모드
* 범위 / LOS / AOE / 비용 정의
* 타일 속성 정의
* AI 가중치 정의

이 레이어는 **런타임 로직이 읽는 설정값**이다.

원칙:

* 데이터는 “무엇을 할 수 있는가”를 정의
* 로직은 “어떻게 처리하는가”를 담당

---

## 2.7 AI Layer

적 행동 결정을 담당한다.

핵심 클래스:

* AIPlanner
* AIProfile

책임:

* 이동 후보 생성
* 스킬 후보 생성
* 타겟 후보 생성
* 후보별 점수 계산
* 최종 행동 선택

AI는 반드시 Rules / Resolution Layer와 같은 규칙을 사용해야 한다.
독립 규칙을 만들면 Preview / Execution / AI 불일치 버그가 발생한다.

---

# 3. 현재 핵심 실행 흐름

현재 전투 흐름은 대략 다음 순서를 따른다.

```plaintext id="icnr55"
BattleController
→ BattleInput / Skill 선택 / Move 계획
→ Confirm
→ GridManager 이동
→ CombatResolver 실행
→ CombatTargetResolver로 대상 판정
→ SkillEffect 적용
→ Hazard / Status / Counter 후처리
→ 턴 종료
```

---

# 4. 핵심 클래스별 역할

## 4.1 BattleController

전투 시스템의 중심 관리자.

주요 책임:

* 턴 순서 구성
* 현재 행동 유닛 관리
* PlannedAction Queue 관리
* Move / Skill 계획 및 Confirm 실행
* EnemyTurnRoutine 실행
* BattleInput과 연결
* TileHighlighter와 연결
* RunSkill 호출
* Hazard와 Status의 턴 타이밍 연결

권장 원칙:

* BattleController는 **결과를 조합**해야지
* 세부 판정 로직이 계속 몰리면 안 된다

즉:

* 타겟 판정은 CombatTargetResolver
* 스킬 해결은 CombatResolver
* 이동 계산은 GridManager
* 상태 처리 핵심은 Unit/StatusEffect

로 보내는 방향이 좋다.

---

## 4.2 BattleInput

플레이어 입력 처리 전용 클래스.

주요 책임:

* 마우스 / 포인터 위치 확인
* hover 타일 갱신
* 클릭 타일 / 클릭 유닛 확인
* Confirm / Cancel 입력 연결
* BattleController에 입력 이벤트 전달

원칙:

* BattleInput은 입력 해석만 한다
* 전투 규칙 판정은 하지 않는다

즉:

* “마우스가 어디를 가리키는가”는 BattleInput
* “그 타일이 유효한가”는 BattleController/Resolver

---

## 4.3 TileHighlighter

전투 하이라이트 / 프리뷰 전용 클래스.

주요 책임:

* Move 타일 표시
* SkillRange 표시
* PreviewArea 표시
* Selected / Actionable / Target / Warning / Hazard FX 표시
* Path / PlannedPath 표시
* Hover Hazard / Planned Hazard 표시
* Ghost / Preview 관련 표시

이 클래스는 **표시 전용**으로 유지하는 것이 중요하다.
판정 로직을 넣기 시작하면 BattleController와 책임이 섞인다.

---

## 4.4 GridManager

맵과 이동의 핵심 관리자.

주요 책임:

* 좌표계 관리
* 점유 관리
* BFS / ReachableData
* 경로 복원
* 이동 루틴
* LOS 판정
* TileView / TileData 조회
* Height 이동 제약

향후 확장 포인트:

* 아군 통과 가능 / 종료 불가
* Height 이동 비용 반영
* 더 정교한 경로 비용 탐색
* 지형 타입별 이동 보정

---

## 4.5 Unit

유닛 개별 상태의 중심 클래스.

주요 책임:

* HP / AP
* 이동 관련 수치
* 상태이상 보관
* 데미지 / 힐 / 실드 처리
* Counter / Invincible 등 상태 질의
* GridPos
* 애니메이션 이벤트 발생

향후 원칙:

* Unit은 가능한 한 “자기 상태”만 관리
* 전투 전체 흐름은 모르도록 유지

---

## 4.6 CombatTargetResolver

전투 대상 판정 전용 클래스.

주요 책임:

* SkillTargetMode별 대상 계산
* Range / LOS 검증
* ClickSingle / ClickTileAOE / Auto / All 계열 처리
* previewPosition 기반 판정

이 클래스는 **전투 규칙 일관성의 핵심**이다.

중요 원칙:

```plaintext id="1uvmvy"
타겟 계산 규칙은
반드시 한 곳에서 관리한다
```

---

## 4.7 CombatResolver

실행 가능한 행동을 실제 결과로 연결하는 클래스.

주요 책임:

* ResolveForExecution
* AP 소모
* 최종 타겟 유효성 확인
* SkillEffect 적용
* Counter 연계

CombatResolver는 “이 스킬을 지금 실제로 실행했을 때 무슨 일이 일어나는가”를 담당한다.

---

## 4.8 HazardSystem

위험 타일 처리 전용 클래스.

주요 책임:

* OnEnter / OnTurnStart / OnTurnEnd 처리
* Burn / Poison / Explosion 처리
* Hazard 결과 반환

향후 확장 포인트:

* Hazard 소모형 타일
* Hazard 확산
* 타일 제거 / 변형
* 상태와 상호작용하는 특수 타일

---

## 4.9 AIPlanner

적 행동 선택 전용 클래스.

주요 책임:

* 이동 후보 계산
* 스킬 후보 계산
* 타겟 후보 계산
* 기대 가치 평가
* 최종 행동 반환

AIPlanner는 **독립 규칙 엔진이 아니라**
공통 Resolver 규칙을 사용하는 “선택기”여야 한다.

---

# 5. 의존성 구조

권장 의존성 방향:

```plaintext id="7ly44j"
Presentation
→ Battle Flow
→ Rules / Resolution
→ World / Unit / Data
```

보다 구체적으로는:

```plaintext id="vqz8h9"
BattleInput      → BattleController
TileHighlighter  → BattleController 결과 표시
BattleController → GridManager / CombatResolver / AIPlanner / Unit
CombatResolver   → CombatTargetResolver / SkillEffect / Unit
CombatTargetResolver → GridManager / Unit / SkillData
AIPlanner        → CombatTargetResolver / GridManager / SkillData / Unit / AIProfile
Unit             → StatusEffect
TileView         → TileData
```

원칙:

* 아래 레이어는 위 레이어를 몰라야 한다
* 위 레이어가 아래 레이어를 조합해야 한다

# 6. 확장성 기준

## 6.1 새 스킬 추가

새 스킬 추가 시 수정 위치:

* SkillData
* SkillEffect
* 필요 시 CombatTargetResolver 타겟 모드 확장
* 필요 시 AIPlanner 평가 로직 확장
* 필요 시 UX 프리뷰 확장

## 6.2 새 상태이상 추가

수정 위치:

* StatusEffect 파생 클래스
* Unit 상태 적용/질의 보강
* AIPlanner 상태 가치 보강
* 월드 FX / HUD 상세 표시 연결

## 6.3 새 타일 기믹 추가

수정 위치:

* TileData
* TileView
* GridManager
* HazardSystem 또는 별도 TerrainRule 시스템
* AIPlanner 위험/선호 계산

## 6.4 새 타겟 패턴 추가

수정 위치:

* SkillTargetMode
* CombatTargetResolver
* TileHighlighter preview 표시
* BattleController hover/lock 연결
* AIPlanner 후보 생성

# 7. 지금 구조에서 중요한 리팩토링 원칙

## 7.1 BattleController 비대화 방지

현재 BattleController는 이미 많은 책임을 갖고 있다.
따라서 앞으로는 아래 성격의 코드를 계속 분리하는 것이 좋다.

분리 후보:

* Preview 계산 보조기
* Fail feedback 전용 도우미
* PlannedAction 관리 헬퍼
* PreviewActor 표시 전용 컴포넌트
* Skill UX 전용 보조 클래스

## 7.2 표시와 판정 분리

절대 섞이면 안 되는 것:

* TileHighlighter ↔ 판정 로직
* UnitHud ↔ 전투 규칙 계산
* BattleInput ↔ 전투 규칙 판정

## 7.3 공통 규칙 단일화

중복되면 안 되는 것:

* Range 판정
* LOS 판정
* Target 계산
* AOE 대상 범위
* Friendly Fire 규칙

이 규칙들은 반드시 Resolver 중심으로 모여야 한다.

# 8. 권장 폴더 구조

현재 파일명을 유지하되, 장기적으로는 다음 구조가 좋다.

```plaintext id="ub36ud"
Scripts/
    Battle/
        BattleController
        BattleInput
        PlannedAction (추후 분리 가능)

    Battle/Resolution/
        CombatResolver
        CombatTargetResolver
        HazardSystem

    Battle/AI/
        AIPlanner
        AIProfile

    Grid/
        GridManager
        TileView
        TileData

    Units/
        Unit
        UnitHud

    Status/
        StatusEffect
        BurnStatus
        PoisonStatus
        StunStatus
        ...

    Skills/
        SkillData
        SkillEffect
        DealDamageEffect
        HealEffect
        ApplyStatusEffectBase
        ...

    UX/
        TileHighlighter
        PreviewActorPresenter (추후)
        BattleFeedbackPresenter (추후)
```

---

# 9. 현재 구조에서 유지해야 할 핵심 원칙

```plaintext id="e6p31f"
1. Preview / Execution / AI는 같은 규칙을 써야 한다
2. BattleController는 오케스트레이터여야 한다
3. GridManager는 맵/이동 책임만 가진다
4. Resolver는 전투 판정 책임을 가진다
5. 표시 클래스는 계산하지 않는다
6. Data는 정의, Logic은 실행을 담당한다
```

---

# 10. 향후 리팩토링 우선순위

## 우선순위 1

* PreviewActor 관련 시각 로직 분리
* ActionQueue 관련 구조 분리
* BattleController 내부 helper 성격 함수 정리

## 우선순위 2

* Skill 타겟 패턴 확장 대비 Resolver 구조 정리
* 상태이상 FX 시스템 분리
* Fail feedback / floating text / sound 분리

## 우선순위 3

* 전투 로그 / 디버그 도구 추가
* AI 디버그 시각화
* 전투 시뮬레이션 테스트 구조 추가

---
